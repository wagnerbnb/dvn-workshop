# IMPL-ADR-0002-2026-05-23

## Reference ADR

`docs/ADR-0002-remote-backend.md` — Remote Backend: S3 com Native Locking para Terraform State

## Summary

Provisioned the `remote-backend` Terraform stack, which creates an S3 bucket configured as a remote state backend for all other stacks in the repository. The implementation uses S3 native locking (`use_lockfile = true`) as decided in ADR-0002, eliminating the need for a DynamoDB table.

## Deviations from ADR

### Minor: output name `s3_bucket_region` added

The ADR specified outputs `s3_bucket_id` and `s3_bucket_arn`. A third output `s3_bucket_region` was added exposing `aws_s3_bucket.state.bucket_region`, which is referenced in the ADR's backend block example (`region = "us-east-1"`). This is additive and consistent with the ADR's intent (outputs expose information needed by consuming stacks).

### MCP finding: `bucket_key_enabled` position

The ADR implementation guideline listed `bucket_key_enabled` under `apply_server_side_encryption_by_default`. Per MCP validation of provider doc 12311377, `bucket_key_enabled` is a sibling of `apply_server_side_encryption_by_default` within the `rule` block — not nested inside it. The implementation follows the correct provider schema.

## Rollback Plan

The `remote-backend` stack state is stored locally (deliberate chicken-and-egg design). Rollback options:

1. **Revert code + apply**: `git revert` the stack creation commit, then `terraform destroy -var-file="envs/production.tfvars"` (requires explicit human confirmation per guardrails).
2. **Pre-condition**: Before destroying the bucket, all consuming stacks must have their state migrated back to local (`terraform init -migrate-state` with a local backend configuration).
3. **State backup**: The local state file at `remote-backend/terraform.tfstate` is the source of truth for this stack and should be committed to version control or stored securely.

## State Migration: 01-networking-stack-ai

### What was done

Added a `backend "s3"` block to `01-networking-stack-ai/versions.tf` and migrated the local state to the remote S3 bucket.

### Backend block added to `01-networking-stack-ai/versions.tf`

```hcl
backend "s3" {
  bucket       = "dvn-workshop-production-terraform-state"
  key          = "networking/terraform.tfstate"
  region       = "us-east-1"
  use_lockfile = true
  encrypt      = true
}
```

### Commands Executed

```bash
cd 01-networking-stack-ai
echo "yes" | terraform init -migrate-state
# Output: Successfully configured the backend "s3"!
#         Terraform has been successfully initialized!

# Verified all 20 resources are accessible from the remote backend

rm terraform.tfstate
# Local state file removed after confirmed successful migration
```

### Migration Verification

`terraform state list` returned all 20 resources managed by the networking stack, confirming state was copied correctly to `s3://dvn-workshop-production-terraform-state/networking/terraform.tfstate`.
